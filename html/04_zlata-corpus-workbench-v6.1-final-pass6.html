User guide ?

A full operating manual for the three-sprint workflow, written to live inside the app.

Quick start

Start in Sprint 1 if you have many files. Open Sprint 2 if you are refining one record. Use Sprint 3 only after the current record is good enough to reuse.

How to avoid weak records

Do not skip setup, consent, summary, and notes. Those fields are the difference between a searchable corpus and a pile of disconnected entries.

How to get the best generator output

Use Faithful Zlata Match first, pull from a strong graded record, build the outline, fix anything generic, and only then generate the full prompt.

Acceptance checklist. Final QA pass active. This pass focuses on final alignment: Track A auto-fill, comparison fields, z-style persistence, generator gating, richer pull-from-record, contextual help, and persistence across reloads.
Purpose. This workbench is designed to help you ingest many stories, convert them into structured records, score them against a Zlata-style grading scheme, attach optional image references, and then reuse the best records as inputs for prompt-based analysis or story generation. The app keeps those stages separate on purpose. Sprint 1 is for intake and triage, Sprint 2 is for careful grading, and Sprint 3 is for reuse. That separation reduces mistakes because you are not trying to classify, score, edit, and generate at the same time.

How the workflow is meant to feel. Start broad, then get precise. Use the Batch Inbox when you have many files and need to create starter records quickly. Move into the Style Grader only after you choose one record to inspect in detail. Save often, because the ranking table and record list are driven by the current saved record state. When the current record is solid, use the Story Generator to reuse its settings, moves, image tags, and tonal observations. The Guide tab is the reference layer that explains what each button does, which path to use for common tasks, and how data moves through the app.

What counts as a record. A record is the core data object in this app. It stores metadata like title, author, source, URL, setup, and consent mode; slider values for discovery, flexibility precision, erotic tension, chemistry, training, and writing quality; text fields like summary and match notes; move vocabulary; and any attached images. The comparison table, record list, prompts, thumbnails, KPIs, and generator reuse all pull from that same record object. In practical terms, if you want information to show up elsewhere in the app, save it into the current record first.

Step-by-step: add a batch of story files. Open Sprint 1 and drop one or more TXT or RTF files into the Batch Inbox. The app reads each file, cleans RTF if needed, calculates an initial word count, sets a default source, and creates one starter record per file. Those new records immediately appear in the queue list and the comparison table. Click a queue item to select it, change its status or source in Quick Record Actions if needed, and then use Open Selected in Grader to move into detailed review.

Step-by-step: grade one story carefully. In Sprint 2, confirm the metadata first: title, author, source, URL, length, POV, setup, consent, and tags. Next, review or paste the story text. Then adjust the sliders based on the actual reading, not on a guess. Tick only the move vocabulary the story truly supports. Add a short summary and a match note that explains why the story is or is not a close Zlata-style fit. Click Save Current when you want the score, tier, comparison table row, and record list to reflect the final state.

Step-by-step: attach and manage images. In the Images panel, drop one or more files into the image intake zone. The app stores each image on the current record and renders an image card. Add a filename correction if needed, then add a caption and comma-separated tags. Use Set as Primary on the strongest image so the comparison table and thumbnail review use the right visual. Click the thumbnail to preview it. Use Remove if an image does not belong to the record. If you want placeholder records for your attached starter images, use Load Starter Attached-Image Record and then upload files with matching names.

Step-by-step: compare many saved stories. Return to Sprint 1 and use the Bulk Multi-Story Comparison table. The table supports filters for status and special flags, plus search by title, author, source, setup, and tags. Click a table header to sort by that field. Use Open to load a record directly in the grader or Select to mark it as the active queue item. The Batch Thumbnail Review underneath shows only the records that match the current filters, which is useful when you want to scan visual references without leaving the filtered comparison state.

Step-by-step: export, import, and keep the corpus portable. In Sprint 2, use Export Corpus JSON to save every record in the session as one file, or use Export Current JSON to save only the active record. JSON import accepts either an array of records or one single record. Imported data is normalized into the app’s expected structure, including tags, images, slider values, and computed score. This is the safest way to preserve work between sessions because the app is a static HTML tool rather than a database-backed system.

Step-by-step: generate a grading prompt. After you finish or partially finish a record, click Generate Grader Prompt. The prompt box is filled with the current record fields, image notes, and the current story text excerpt. This is useful when you want an external assistant such as Perplexity Computer to help refine summaries, tags, image captions, or scoring rationale. Use Copy Prompt to send it out. When the assistant returns structured output, paste the refined information back into the relevant fields and save again.

Step-by-step: generate a new story prompt from a record. Open Sprint 3 and click Pull from Current Record. The generator fills core fields from the current graded record, including likely male point of entry, flexible heroine framing, scenario, tone, move vocabulary, chemistry notes, and image context. Adjust any fields you want to change, then click Generate Story Prompt. The output is designed for drafting tools and keeps the same tonal rules you established in the grader. This makes Sprint 3 a reuse layer rather than a disconnected toy.

Data flow in plain language. Files come into the app through story or image intake. Intake creates or enriches a record. The current record then feeds scoring, interpretation, KPIs, the record list, the comparison table, the thumbnail board, the grading prompt, and the story generator. No separate hidden record is used for those sections. That means nearly every visible output in the interface depends on the current record being saved and normalized correctly.

Quality control and safe habits. Use Batch Inbox for rough intake, but do not trust starter values as final judgments. Save after meaningful edits. Treat the Setup and Consent fields as high-leverage because they can materially change score and tier. Use notes to record why a score is high or low, not just what happened in the story. For images, make sure the primary flag is set correctly because the table and thumbnail view rely on it. If something looks wrong elsewhere in the app, first confirm you are editing the record you think you are editing, then save again.

Troubleshooting. If the comparison table seems stale, save the current record and check your filters. If the generator has weak context, pull from the current record again after saving. If a story imports as unreadable text, use Clean Story Text to strip RTF control noise. If a copied prompt is empty, regenerate it first. If an image thumbnail is missing, the record may only contain a placeholder filename rather than actual uploaded image data. In that case, upload the matching image file again.

Data flow ?

How information moves through the app from intake to reuse.

1. Input layer

TXT and RTF files enter through Batch Inbox or Story Text Intake. Images enter through the image intake dropzone. Manual edits can also start directly in the grader.

↓
2. Record layer

The app converts intake into one normalized record object. This object stores metadata, story text, slider scores, tags, moves, notes, images, and timestamps.

↓
3. Scoring layer

The weighted scoring model reads the current record and computes score and tier. Consent and setup can add or subtract from the weighted base.

↓
4. Display layer

Saved record data populates KPIs, live interpretation, record list, sortable comparison table, and batch thumbnail review.

↓
5. Reuse layer

The same record powers the grader prompt, JSON exports, generator autofill, and image-context summaries for drafting workflows.

First-run walkthrough

How to use the workbench in the right order

This onboarding is designed to keep the workflow simple: intake first, judgment second, reuse third.

Step 1 · Batch Inbox

Upload story files to create starter records quickly.

Step 2 · Grader

Open one selected record, refine metadata, score the rubric, and save.

Step 3 · Images

Attach images, add captions, and choose a primary thumbnail if visual context matters.

Step 4 · Compare

Use the comparison table and filters to rank many stories at once.

Step 5 · Reuse

Export JSON, generate grader prompts, or pull a graded record into the story generator.

Recommended first action

Upload 2 to 5 files in Sprint 1 so you have enough material to learn the queue and comparison views.

Common mistake to avoid

Do not score a record before checking setup, consent, and text quality. Those fields shape the interpretation of every score.

Fast path

Sprint 1 → select record → open in grader → save → return to table → pull into generator if needed.
